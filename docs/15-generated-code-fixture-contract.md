# Generated-Code Fixture Contract Seed

This document defines how GrOS may represent future generated-code fixtures before a real `.grw` compiler exists. It is a test contract seed only. It does not add a parser, compiler, interpreter, linker, code generator, allocator, hosted-native executable output, executable `.gwo` loader, or boot banner change.

## Purpose

The ABI stability gate requires at least one generated-code fixture before `.grw` lowering can start. This seed defines the shape of that fixture without pretending that generated code exists today.

The current repository can validate:

- raw `.gwn` source
- `.gwo` byte artifacts
- runtime ABI byte fixtures
- real16 memory model fixtures
- stage-2 smoke start

It cannot yet validate compiler output because no `.grw` compiler exists.

## Fixture Meaning

A generated-code fixture has two meanings:

```txt
expected behavior contract
expected low-level representation
```

Until real `.grw` tooling exists, a fixture may contain a handwritten expected `.gwn` representation and a committed expected `.gwo` artifact. That pair is a future compiler target, not compiler output.

The repository must describe such fixtures as:

```txt
expected generated-code fixture
```

It must not describe them as:

```txt
compiled from `.grw`
produced by the `.grw` compiler
generated by current tooling
```

## Allowed Fixture Shape

The first fixture directory shape is reserved as:

```txt
fixtures/generated-code/<fixture-name>/
```

Reserved files:

```txt
source.grw
expected.gwn
expected.gwo
manifest.txt
```

`source.grw`:

```txt
Informational future source input.
Not parsed by the current build.
Not compiled by the current build.
```

`expected.gwn`:

```txt
Handwritten low-level expected representation.
The current Bash build may assemble this with scripts/gwnraw.sh.
The raw boundary name must stay profile-specific for the fixture family.
```

`expected.gwo`:

```txt
Committed golden bytes built from expected.gwn.
Not claimed as `.grw` compiler output.
```

`manifest.txt`:

```txt
ASCII key-value metadata.
One key per line.
No JSON, YAML, TOML, Python, or generated metadata format is required.
```

Seed manifest keys:

```txt
name=<fixture-name>
profile=gros.x86.bios.real16.stage2.v0
source=source.grw
expected_gwn=expected.gwn
expected_gwo=expected.gwo
expected_size=2048
entry_address=0000:8000
status=expected-only
```

The only valid seed status is:

```txt
expected-only
```

## Validation Contract

The Bash-only validation script is:

```txt
scripts/check_generated_fixtures.sh
```

It validates existing fixtures by:

- checking the manifest keys
- checking the expected byte size
- checking the expected entry address against the low-level `.gwn` origin
- checking that the manifest profile and `.grw` target declaration match
- checking that the `.gwn` raw boundary stays in the current real16 generated fixture family
- rejecting unknown fixture status values
- building `expected.gwn` through `scripts/gwnraw.sh`
- comparing the built bytes with `expected.gwo`
- checking profile-specific ABI fixtures over `expected.gwo`
- confirming that `source.grw` is informational only

The validation script passes when no generated-code fixtures exist yet. That keeps the contract active before the first fixture is added.

The validation script must not:

- parse `.grw`
- compile `.grw`
- infer `.grw` types
- lower `.grw` to `.gwn`
- invoke Python, Node.js, Rust, Go, C, or another hosted implementation language
- claim that `source.grw` produced `expected.gwo`

## Minimal Future Source Shape

The first informational `source.grw` may use the current front-end seed:

```grw
target "gros.x86.bios.real16.stage2.v0"

fn main() -> void {
    return;
}
```

The corresponding `expected.gwn` remains handwritten until the ABI stability gate opens.

For the first current-profile fixture, the raw boundary uses this family:

```gr
raw x86.bios.real16.generated.minimal_main_void {
    origin 8000
}
```

The `generated` segment marks this as an expected generated-code fixture family,
not as current compiler output.

## Relationship To The ABI Gate

The first expected-only fixture lives at:

```txt
fixtures/generated-code/minimal-main-void/
```

It proves the fixture representation and `.gwn` to `.gwo` parity path. It is still not compiler output.

Its minimal `fn main()` runtime contract is defined in:

```txt
docs/16-grown-main-runtime-contract.md
```

The ABI gate still stays closed until at least one fixture exists and passes local validation without adding a high-level build dependency.

## Non-Goals

This seed does not define:

- `.grw` parsing
- `.grw` type checking
- `.grw` lowering
- `.gwn` emission from `.grw`
- object files
- relocation records
- symbol tables
- linking
- generated-code optimization
- hosted-native executable output
- a new `.gwo` payload loader
- a version bump
