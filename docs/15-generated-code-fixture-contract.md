# Generated-Code Fixture Contract Seed

This document defines how GrOS may represent future generated-code fixtures before a real `.gn` compiler exists. It is a test contract seed only. It does not add a parser, compiler, interpreter, linker, code generator, allocator, hosted-native executable output, executable `.gro` loader, or boot banner change.

## Purpose

The ABI stability gate requires at least one generated-code fixture before `.gn` lowering can start. This seed defines the shape of that fixture without pretending that generated code exists today.

The current repository can validate:

- raw `.gr` source
- `.gro` byte artifacts
- runtime ABI byte fixtures
- real16 memory model fixtures
- stage-2 smoke start

It cannot yet validate compiler output because no `.gn` compiler exists.

## Fixture Meaning

A generated-code fixture has two meanings:

```txt
expected behavior contract
expected low-level representation
```

Until real `.gn` tooling exists, a fixture may contain a handwritten expected `.gr` representation and a committed expected `.gro` artifact. That pair is a future compiler target, not compiler output.

The repository must describe such fixtures as:

```txt
expected generated-code fixture
```

It must not describe them as:

```txt
compiled from `.gn`
produced by the `.gn` compiler
generated by current tooling
```

## Allowed Fixture Shape

The first fixture directory shape is reserved as:

```txt
fixtures/generated-code/<fixture-name>/
```

Reserved files:

```txt
source.gn
expected.gr
expected.gro
manifest.txt
```

`source.gn`:

```txt
Informational future source input.
Not parsed by the current build.
Not compiled by the current build.
```

`expected.gr`:

```txt
Handwritten low-level expected representation.
The current Bash build may assemble this with scripts/grraw.sh.
```

`expected.gro`:

```txt
Committed golden bytes built from expected.gr.
Not claimed as `.gn` compiler output.
```

`manifest.txt`:

```txt
ASCII key-value metadata.
One key per line.
No JSON, YAML, TOML, Python, or generated metadata format is required.
```

Seed manifest keys:

```txt
name=<fixture-name>
profile=gros.x86.bios.real16.stage2.v0
source=source.gn
expected_gr=expected.gr
expected_gro=expected.gro
status=expected-only
```

The only valid seed status is:

```txt
expected-only
```

## Validation Contract

A future Bash-only validation script may validate a fixture by:

- checking the manifest keys
- rejecting unknown fixture status values
- building `expected.gr` through `scripts/grraw.sh`
- comparing the built bytes with `expected.gro`
- checking profile-specific ABI fixtures over `expected.gro`
- confirming that `source.gn` is informational only

The validation script must not:

- parse `.gn`
- compile `.gn`
- infer `.gn` types
- lower `.gn` to `.gr`
- invoke Python, Node.js, Rust, Go, C, or another hosted implementation language
- claim that `source.gn` produced `expected.gro`

## Minimal Future Source Shape

The first informational `source.gn` may use the current front-end seed:

```gn
target "gros.x86.bios.real16.stage2.v0"

fn main() -> void {
    return;
}
```

The corresponding `expected.gr` remains handwritten until the ABI stability gate opens.

## Relationship To The ABI Gate

This contract does not satisfy the generated-code fixture requirement by itself. It only defines how the requirement must be represented when the first fixture is added.

The ABI gate still stays closed until at least one fixture exists and passes local validation without adding a high-level build dependency.

## Non-Goals

This seed does not define:

- `.gn` parsing
- `.gn` type checking
- `.gn` lowering
- `.gr` emission from `.gn`
- object files
- relocation records
- symbol tables
- linking
- generated-code optimization
- hosted-native executable output
- a new `.gro` payload loader
- a version bump
